Why Graph Analytics Changes Everything

Traditional relational databases are exceptional at what they were designed for: fast lookups, aggregations, and transactional integrity. But detecting fraud in a financial network demands something fundamentally different. It demands the ability to ask questions like: Does money flow from account A back to itself through a chain of accounts? In SQL, answering that requires recursive common table expressions that grow exponentially in complexity with each additional hop.

Graph databases flip that challenge on its head. In a property graph, accounts become nodes and money transfers become directed edges. A circular payment chain is then just a path in that graph that starts and ends at the same node. What would take pages of SQL becomes a few elegant lines of PGQL.

🔵

Nodes (Vertices)

Each bank account is represented as a node. Properties like account ID and balance are attached directly to the node.

➡️

Edges (Relationships)

Every fund transfer is a directed edge. The direction carries meaning: who sent money to whom.

🏷️

Properties

Both nodes and edges carry attributes: transaction amounts, timestamps, risk scores, and computed metrics like PageRank.

🔄

Circular Chains

A cycle in the graph where funds eventually return to the origin account, a hallmark of money laundering and layering schemes.

Loading the Graph into Oracle PGX

The first step is bringing the graph into Oracle's in-memory graph server, Oracle PGX. PGX is designed for high-throughput parallel graph analysis and makes multi-hop traversals dramatically faster than disk-based approaches. The Python-PGX API makes this simple:

Python · PGX 
# Load BANK_GRAPH into the in-memory graph server
GRAPH_NAME = "BANK_GRAPH"

graph = session.get_graph(GRAPH_NAME)

if graph == None:
    session.read_graph_by_name(GRAPH_NAME, "pg_view")
    print("Graph " + GRAPH_NAME + " successfully loaded")
    graph = session.get_graph(GRAPH_NAME)
else:
    print("Graph '" + GRAPH_NAME + "' already loaded")
OUTPUT
Graph 'BANK_GRAPH' already loaded

With the graph in memory, every subsequent algorithm and query operates at in-memory speeds, which matters enormously when traversing thousands of interconnected accounts.

Querying the Graph with PGQL

PGQL (Property Graph Query Language) is Oracle's SQL-like language built specifically for property graphs. It extends familiar SQL concepts with pattern-matching syntax that describes graph structures naturally. The core of any PGQL query is the MATCH clause:

PGQL 
/* Query and visualize 100 elements of BANK_GRAPH */
SELECT *
FROM MATCH (s)-[t]->(d) ON bank_graph
LIMIT 100

Here, (s) is the source account node, [t] is the transaction edge, and (d) is the destination account node. The arrow direction captures the flow of money. What renders in the notebook is not a flat table but an interactive force-directed graph where you can explore connections visually.

Oracle Graph Studio · BANK_GRAPH · 100 Elements
SELECT * FROM MATCH (s)-[t]->(d) ON bank_graph LIMIT 100  ·  Force-directed layout · Nodes = accounts · Edges = transfers
💡
PGQL also supports runtime bind parameters, allowing dynamic traversal at query time. By passing a specific account ID as a parameter, the query filters the graph traversal to that account's immediate neighborhood, which is exactly how fraud investigators drill into specific suspects.

Finding the Most Connected Accounts

Before hunting for cycles, I identified the accounts with the highest total transaction volume using PGQL's built-in IN_DEGREE and OUT_DEGREE functions, which count incoming and outgoing edges respectively:

PGQL 
/* Top 10 accounts by total transaction count */
SELECT a.acct_id,
       (in_degree(a) + out_degree(a)) AS num_transactions
FROM MATCH (a) ON bank_graph
ORDER BY num_transactions DESC
LIMIT 10

Two accounts immediately stood out: 387 and 934. Account 934 had 44 total transactions, placing it firmly in the top tier of activity. This high connectivity alone is not evidence of fraud, but it is exactly the kind of signal that warrants deeper investigation.

Circular Chain Detection: The Core Use Case

This is where graph analytics earns its place in the fraud detection toolkit. I queried for circular payment chains of exactly 4 hops that originate and terminate at account 934:

PGQL · Cycle Detection 
/* Circular payment chains of length 4 from account 934 */
SELECT v, e, v2
FROM MATCH (v)-[e]->{4}(v2) ON bank_graph
WHERE v.acct_id = 934
  AND v2.acct_id = 934
Circular Chains · Length 4 · Account 934
MATCH ALL (a)-[:TRANSFERS]->{4}(b) WHERE a.acct_id=934 AND id(a)=id(b)  ·  3 circular chains detected  ·  Account 934 highlighted in red
Circular Chains · Length 5 · Account 934
MATCH ALL (a)-[:TRANSFERS]->{5}(b) WHERE a.acct_id=934 AND id(a)=id(b)  ·  More cycles detected at 5 hops — elevated suspicion signal
INVESTIGATION RESULTS
1
Account 934 has 3 distinct circular payment chains of length 4. Money leaves the account, passes through three intermediaries, and returns to the origin.
2
Extending the search to 5-hop chains reveals even more cycles, dramatically increasing the suspicion level. The number of circular paths is directly proportional to the risk signal.
3
Account 387 displays similar behavior: high degree centrality and circular payment patterns placing it firmly in the suspect list alongside 934.

In a relational model, detecting a 5-hop circular chain requires five nested self-joins. In PGQL on a property graph, it is a single pattern match that runs in seconds against an in-memory graph.

Applying Graph Algorithms at Scale

Oracle Graph Studio ships with over 80 built-in graph algorithms accessible directly from the Python-PGX API. Here is how I layered multiple algorithms to build a complete picture of the suspicious activity.

Graph Analytics + AI
As enterprise AI systems become more relationship-driven, graph analytics is becoming a critical foundation for intelligent applications. Oracle Graph Studio combines graph querying, graph machine learning, and in-memory analytics within Oracle Autonomous AI Database, allowing organizations to build connected AI workflows directly on top of transactional data.

By transforming relational data into property graphs, businesses can uncover hidden relationships, detect fraud patterns, power recommendation systems, build knowledge graphs, and improve AI reasoning across highly connected datasets. Algorithms like DeepWalk and PageRank make it possible to move beyond static rule-based analysis into structurally aware intelligence systems.
Algorithm 01
PageRank

Measures the global importance of each account node. Accounts with high PageRank are either heavily transacted or deeply connected to accounts that are. Account 934 ranked in the top 5 globally.

Algorithm 02
Personalized PageRank

Computes PageRank relative to a specific seed set. Using accounts 934 and 387 as seeds exposed account 406 as the next highest-risk account in the network.

Algorithm 03
ShortestPathHopDist

Calculates the minimum hop distance from every account to the suspect nodes. Accounts within 1 to 2 hops carry significantly elevated risk of involvement.

Algorithm 04
DeepWalk (Graph ML)

A graph embedding model that learns structural similarity between nodes. Accounts that behave structurally like 934 and 387 were surfaced without any explicit rule definition.

PageRank: Quantifying Influence

PageRank was originally developed to rank web pages, but its logic translates perfectly to financial networks: a node is important if it receives flows from other important nodes. Running it on the banking graph:

Python · PGX 
graph = session.get_graph("BANK_GRAPH")
analyst.pagerank(graph)
OUTPUT
VertexProperty(name: pagerank, type: double, graph: BANK_GRAPH)

The result is a new property, pagerank, attached to every account node. Querying the top accounts reveals that 934 ranks in the top 5, and 387 tops the list. The PageRank metric independently corroborates what circular chain detection already suggested.

Personalized PageRank: Mapping the Risk Radius

Once the fraud team flagged 934 and 387 as suspects, I used Personalized PageRank to measure how closely every other account is connected to this specific pair:

Python · PGX 
vertices = graph.create_vertex_set()
vertices.add_all_by_label("BANK_ACCOUNTS")
analyst.personalized_pagerank(graph, vertices)
OUTPUT
VertexProperty(name: personalized_pagerank, type: double, graph: BANK_GRAPH)

The results placed account 406 immediately after the two known suspects, meaning it sits structurally close to both 934 and 387 in the transaction network. When I queried its immediate neighbors, the connectivity confirmed elevated risk. This is investigative graph analytics working exactly as it should.

Graph Machine Learning with DeepWalk

The final layer of analysis moved beyond hand-crafted queries into learned graph representations. DeepWalk is a graph embedding technique that treats random walks through the graph as sentences and learns vector representations of nodes using a word2vec-style model. Nodes that appear in similar structural contexts end up close in the embedding space.

Python · PGX · DeepWalk 
# Build and train the DeepWalk model
model = analyst.deepwalk_builder(
    min_word_frequency=1,
    batch_size=512,
    num_epochs=1,
    layer_size=128,
    walks_per_vertex=6,
    walk_length=5,
    window_size=2
)

model.fit(graph)
# Training completed in 42.39 seconds

No rule was written for this discovery. The model learned it.
