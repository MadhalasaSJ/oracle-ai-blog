Fraud rarely announces itself as a single bad transaction. It shows up as a pattern across geography and time, hidden inside what looks like perfectly normal activity. Traditional rule engines cannot see that pattern. This pipeline can.

I have been thinking for a while about what it would take to build a fraud detection system that genuinely goes beyond thresholds and blacklists. The problem with rule-based approaches is that they evaluate each transaction in isolation. They do not ask: where was this customer an hour ago? Are they typically in this region at all? Does this transaction fit the behavioral footprint they have established over the past month?

To answer those kinds of questions, you need to treat transactions as spatiotemporal events, not just dollar amounts. That is the core idea behind what I built here: a pipeline that clusters customer transaction behavior across both space and time, then identifies transactions that fall outside those clusters in a statistically meaningful way.

How the System is Put Together

The whole thing runs on Oracle Cloud Infrastructure, with a clean split between what happens inside the database and what happens in Python. Oracle handles the heavy geospatial SQL. Python handles the clustering algorithm. Folium handles the maps. Each layer does exactly what it is best at.

One thing I was deliberate about: keeping the geospatial operations inside Oracle rather than pulling raw coordinates into Python and doing the geometry work there. When you have a spatial index in place and run SDO_WITHIN_DISTANCE inside the database, you get the full benefit of that index. Pull the data out first and you lose that entirely. At scale, that difference is significant.

Getting the Data Ready

The data model is two tables. locations holds longitude and latitude for each transaction site. transactions records each event linked to a location and a customer. Simple enough, but enough to represent real financial behavior patterns across geography and time.

Projecting coordinates into meters

Raw longitude and latitude coordinates are not usable for distance math in the Python libraries I needed. Everything has to be in meters. So the first step was a database function that converts coordinates into Web Mercator (EPSG:3857), giving x and y values in meters.

Oracle SQL — lonlat_to_proj_geom function
CREATE OR REPLACE FUNCTION lonlat_to_proj_geom (
    longitude IN NUMBER, latitude IN NUMBER)
RETURN SDO_GEOMETRY DETERMINISTIC IS
BEGIN
  IF latitude IS NULL OR longitude IS NULL
  THEN RETURN NULL;
  ELSE
    RETURN sdo_cs.transform(
      SDO_GEOMETRY(2001, 4326,
                   sdo_point_type(longitude, latitude, NULL),
                   NULL, NULL),
      3857);  -- Web Mercator: x/y in meters
  END IF;
END;

Rather than adding a new spatial column to the table (expensive for large schemas), I used Oracle's function-based spatial index pattern. You create an index on the function itself. The database then treats calls to that function exactly like a normal indexed column. All spatial operations benefit from the index automatically, with zero schema overhead.

Converting dates to epoch integers

ST-DBSCAN works on numerical arrays. Oracle DATE types do not feed into numpy cleanly, so I added an epoch column to convert transaction dates into Unix timestamps, that is, seconds since January 1, 1970.

Oracle SQL — adding epoch time
ALTER TABLE transactions ADD (trans_epoch_date integer);

UPDATE transactions
SET trans_epoch_date = (trans_date - date'1970-01-01') * 86400;

The Clustering Logic: ST-DBSCAN

This is where the machine learning piece lives. I used ST-DBSCAN, a spatiotemporal extension of the density-based DBSCAN algorithm. The key difference from standard DBSCAN is that it clusters points based on two proximity conditions at the same time: they have to be close in space and close in time. A transaction in New York and a transaction in London might be geographically distant but occur in the same month, and ST-DBSCAN handles that correctly.

Why not standard DBSCAN?

Standard DBSCAN would cluster all of a customer's New York transactions together regardless of when they happened. A customer active in New York from January to March, then again from September to November, might have two separate behavioral periods. ST-DBSCAN separates them because it also enforces a time window, giving you much more precise behavioral clusters to work with.

Three parameters define how the clustering behaves:

1

eps1: 5,000 meters spatial radius

Transactions within 5 km of each other count as geographically co-located. This covers a typical city or suburban zone without merging genuinely separate areas.

2

eps2: 3,000,000 seconds (roughly 35 days)

Transactions within about a month of each other are treated as temporally adjacent. This captures regular monthly patterns while still isolating one-off events that fall outside a cluster's active window.

3

min_samples: 5 transactions

A neighborhood needs at least 5 transactions to qualify as a legitimate cluster. Anything below that gets labeled -1 (noise), which makes it a candidate for anomaly investigation downstream.

The Python code pulls each customer's transactions from Oracle, reshapes them into a numpy array with columns for time, x (meters), and y (meters), then runs ST-DBSCAN.

Python — ST-DBSCAN clustering per customer
# Pull transaction geometries projected into meters
cursor.execute("""
    SELECT a.cust_id, a.trans_id, a.trans_epoch_date,
           (lonlat_to_proj_geom(b.lon,b.lat)).get_wkt()
    FROM transactions a, locations b
    WHERE a.location_id=b.location_id
    AND cust_id=:cust""", cust=cust)

gdf = gpd.GeoDataFrame(cursor.fetchall(),
    columns=['cust_id', 'trans_id', 'epoch_date', 'geometry'])
gdf['geometry'] = shapely.from_wkt(gdf['geometry'])

# Reshape into [time, x_meters, y_meters] numpy array
df = pd.DataFrame({
    'time': gdf.epoch_date,
    'x': gdf.geometry.x,
    'y': gdf.geometry.y,
    'trans_id': gdf.trans_id
})
data = np.int_(df.values)

# label >= 0 is a cluster ID; label == -1 is noise
st_cluster = ST_DBSCAN(eps1=5000, eps2=3000000, min_samples=5)
st_cluster.fit(data)

Once clustering is done, the labels go back into Oracle in a transaction_labels table. That join is what lets the anomaly detection SQL reference cluster membership as part of the geospatial analysis.

Finding the Anomalies with Spatial SQL

After clustering, I needed to find transactions that were labeled as noise (label = -1) but fell within the active time window of an established cluster, and were more than 500 km away from that cluster's geographic centroid. Those are the suspicious ones.

// ORACLE SPATIAL FUNCTIONS USED

Three spatial operations drive the anomaly logic:

  • SDO_AGGR_CENTROID computes the geographic center of each transaction cluster, representing the customer's behavioral home zone for that time period
  • SDO_WITHIN_DISTANCE efficiently filters out noise transactions that fall within 500 km of a cluster centroid, since those are not suspicious
  • sdo_geom.sdo_distance calculates the exact distance in kilometers between each flagged transaction and the relevant cluster centroid
Oracle SQL — anomaly detection query
WITH
  x AS (
      SELECT a.cust_id, a.trans_id, a.trans_epoch_date,
             lonlat_to_proj_geom(b.lon, b.lat) AS proj_geom, c.label
      FROM transactions a, locations b, transaction_labels c
      WHERE a.location_id = b.location_id
      AND   a.trans_id = c.trans_id),
  y AS (
      SELECT label,
             min(trans_epoch_date) AS min_time,
             max(trans_epoch_date) AS max_time,
             SDO_AGGR_CENTROID(
               SDOAGGRTYPE(lonlat_to_proj_geom(b.lon, b.lat), 0.005)
             ) AS proj_geom
      FROM transactions a, locations b, transaction_labels c
      WHERE a.location_id = b.location_id
      AND   a.trans_id = c.trans_id
      AND   c.label != -1
      GROUP BY label)
SELECT x.cust_id, x.trans_epoch_date,
       (x.proj_geom).get_wkt(), x.trans_id,
       round(sdo_geom.sdo_distance(x.proj_geom, y.proj_geom,
                                   0.05, 'unit=KM'))
FROM x, y
WHERE x.trans_epoch_date BETWEEN y.min_time AND y.max_time
AND   x.label != y.label
AND   x.label = -1
AND   sdo_within_distance(x.proj_geom, y.proj_geom,
        'distance=500 unit=KM') = 'FALSE'

What I like about this query is the combination of conditions. It is not just "this transaction is far from the usual area." It also has to fall within the active time window of a cluster. So if a customer normally transacts in Boston during March and there is a transaction in Tokyo also in March, that gets flagged. The time dimension matters as much as the distance.

What this catches that rules miss

A customer whose behavioral cluster sits around Chicago suddenly has a transaction in Miami during the same week. A rule engine sees a normal transaction. This system sees a noise-labeled point 2,200 km from an active cluster centroid, occurring within that cluster's time window. It gets flagged immediately, with the exact distance recorded.

Closing Thoughts

What surprised me most while building this was how much analytical leverage you get from combining spatial SQL with clustering, compared to either approach on its own. Oracle Spatial by itself can tell you that a transaction is far from a reference point. ST-DBSCAN by itself can tell you that a transaction is outside a behavioral cluster. But together, they let you ask a much sharper question: is this transaction outside a cluster that was specifically active at this time, and is it far enough away that no reasonable explanation covers the distance?

Key design decision worth keeping in mind

Doing spatial aggregation and distance filtering inside Oracle means the spatial index is always working for you. If you pull geometries into Python first and compute distances in GeoPandas, you skip the index entirely. For small datasets that probably does not matter. For anything at production scale, it absolutely does.

The pipeline handles multiple customers cleanly, with per-customer clustering running through three simple function calls: get_cluster_centroids(cust), get_anomalies(cust), and get_map(). Scaling it further would mean parallelizing the clustering loop across customers. But the core architecture, Oracle handling the geospatial SQL and Python handling the clustering math, would stay exactly the same.

If you are working on fraud detection, anomaly detection in mobility data, or any problem where behavior unfolds across both space and time, the spatiotemporal framing is worth exploring. Static rules will always have blind spots. Behavioral clustering grounded in real geographic context makes those blind spots much smaller.
