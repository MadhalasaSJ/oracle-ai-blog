Everything You Need to Know About Oracle Object Storage

What buckets really are, how IAM plugs into everything, and the features most people only discover when they actually need them.

M
Madhalasa SJ
Cloud Explorer
May 13, 2026 · 7 min read

Object Storage is one of those OCI services that sounds simpler than it is. On the surface: buckets, objects, upload, done. But there is real depth in how it handles security, data durability, access patterns, and automation, and that depth is what makes it genuinely useful beyond the "I just need somewhere to dump files" use case.

This is not a tutorial walkthrough. It is an explanation of the product itself, what it is actually doing, what decisions you need to make when you use it, and where the edge cases hide. Everything here is pulled from the official Oracle documentation with extra context filled in where the docs sometimes gloss over things.

What Object Storage Actually Is

Oracle Object Storage is a regional, internet-scale service for storing unstructured data. Files, images, backups, logs, video, raw datasets, anything you can put in a file. The key word there is regional: every bucket you create lives in a specific OCI region, and that is where your data physically sits. It is not replicated globally by default. If you need copies in multiple regions, you set that up explicitly using a replication policy.

The service is structured around a three-level hierarchy: namespace, bucket, object. Your namespace is the top-level identifier that OCI assigns to your tenancy. It is unique, system-generated, and shared across all compartments within a region. You cannot change it, and you will see it in every API endpoint and pre-authenticated URL you ever generate. Below that sit your buckets, and inside buckets live your objects.

One hard constraint worth knowing early

Buckets cannot be nested. You cannot put a bucket inside a bucket. The hierarchy is flat: namespace, then buckets, then objects. If you need a folder-like structure inside a bucket, you do it by convention using / characters in object names to simulate paths, though Object Storage itself does not treat those as real directories.

Buckets: What You Are Configuring and Why It Matters

When you create a bucket, you are locking in a few decisions that you will not be able to change later. The most consequential ones are the storage tier and encryption key. Understanding these upfront saves you from having to delete and recreate buckets later.

Naming

OCI auto-generates bucket names in a format like bucket-20260513-1034. You can rename these, but the rules are stricter than they look: 1 to 256 characters, letters, numbers, hyphens, underscores, and periods only. The thing that trips people up is that bucket names are case-sensitive. Oracle treats media-assets and Media-Assets as two completely different buckets. If you are building any kind of automation that references bucket names, standardize your naming convention from the very beginning.

Storage Tier

Every bucket has a default storage tier, and every object you upload into that bucket inherits it. The tier is set at bucket creation and cannot be changed afterward, so pick carefully.

Standard Tier

Fast and frequent access

Full-speed read/write, no retrieval delay, no minimum storage period. This is the default tier and the right choice for active data, anything you are reading or writing more than a few times a month.

Archive Tier

Long-term, infrequent access

Substantially cheaper per GB, but objects must be explicitly restored before you can download them. Designed for data you are keeping for compliance or disaster recovery but do not expect to touch regularly. A minimum storage period of 90 days applies.

If you have active data that ages into cold data over time, say application logs that you need for a month and then archive for 12 months, OCI's Auto-Tiering feature handles this automatically. You enable it on a Standard tier bucket and OCI will move objects to Archive tier once they have not been accessed for 30 days. You can also configure this manually using lifecycle policies if you want more control over timing and conditions.

Security and Access Control

Object Storage does not have its own access control system. Security runs through OCI's Identity and Access Management (IAM), the same policy engine that controls everything else in OCI. If you are already familiar with IAM, this will feel natural. If you are not, this is the part that trips up most beginners.

Every operation on a bucket or its objects, creating, reading, writing, deleting, requires the user or service to be a member of a group that has an IAM policy granting the corresponding permission. The policy that covers full bucket and object management looks like this:

IAM Policy Reference
Create bucket
BUCKET_CREATE permission required
Change visibility
BUCKET_UPDATE permission required
Full admin access
Allow group ObjectAdmins to manage object-family in tenancy

This matters especially when you are working in multi-team or enterprise environments where compartments are used to isolate resources. Your bucket lives in one compartment and your IAM policy scopes access to that compartment. Someone with access to a different compartment cannot touch your buckets unless a policy explicitly grants it.

Security Zones

If your organization is using OCI Security Zones, which enforce baseline security policies automatically, there are a few hard constraints on Object Storage that you need to know about. Buckets in a security zone cannot be made public. They also must use customer-managed encryption keys from the OCI Vault service rather than the default Oracle-managed keys. And you cannot move a bucket from a security zone to a compartment outside it. These are not optional rules you can override. The platform enforces them at the operation level.

Bucket Visibility: Private, Public, and Everything In Between

When you create a bucket, it defaults to Private. In that state, every single request to read or list its contents must be authenticated and authorized through IAM. That is the right default and the one you should leave in place unless you have a specific reason to change it.

Making a bucket Public means anonymous, unauthenticated access. Anyone with the bucket's URL can read objects, and depending on how you configure it, they can also list all the objects in the bucket. That is a significant exposure for most use cases. Oracle's own documentation explicitly recommends against public buckets for most scenarios and instead points people toward pre-authenticated requests as the better-controlled alternative.

Before you ever make a bucket public

Think carefully. When a bucket is public with list access enabled, anyone can enumerate every object in it. If you have uploaded anything you did not intend to expose, configuration files, internal assets, old backups, they are all visible. If you genuinely need public access, configure it to allow download only and not list, so people can fetch specific objects if they know the URL, but cannot browse the bucket contents.

Pre-Authenticated Requests (PARs)

Pre-authenticated requests are how you share specific objects or an entire bucket with people or systems that do not have OCI credentials. You generate a signed URL with a fixed expiration date, a defined access level (read-only, write-only, read/write), and optionally a specific object. Anyone with that URL can interact with the resource until the URL expires. No login required.

A PAR can be scoped to a single object (the most common use case: here is a download link for this file, valid for 48 hours) or to all objects in a bucket (useful for systems that need to upload or pull files without OCI authentication, like a backup tool running on an on-premises server).

One thing the documentation is very direct about: the full PAR URL is shown exactly once, at the moment of creation. Oracle stores only a hash of it, not the URL itself, so there is literally no way to retrieve it again after you close the creation dialog. Copy it immediately and store it somewhere before clicking away.

A note on PAR URL formats

Oracle has been rolling out a new dedicated endpoint format for pre-authenticated requests to replace the older path-based URL format. If you create a PAR today and see a deprecation warning in the console, that is what it is referring to. Both formats work and the functionality is identical. The new format uses virtual-hosted-style URLs. You may want to update any hardcoded URLs in existing applications before Oracle eventually retires the old format.

The Features That Do the Heavy Lifting

Beyond basic bucket and object management, Object Storage has a set of features that handle data protection, automation, and governance. Most people discover these when they need them, which is often too late if they have not set them up in advance.

Core Protection

Object Versioning

Keeps previous versions of objects whenever files are overwritten or deleted, making accidental data loss recoverable and creating a lightweight audit trail automatically.

Automation

Lifecycle Policies

Automatically archives or deletes objects based on bucket-level rules.

Compliance

Retention Rules

Prevents modification or deletion until the retention period expires.

Infrastructure

Cross-Region Replication

Replicates objects across OCI regions for redundancy, disaster recovery, and lower-latency access patterns.

Event System

Event-Driven Automation

Triggers OCI Functions, Notifications, and workflows whenever objects are uploaded, modified, or deleted.

Security

Customer-Managed Encryption

Uses OCI Vault keys instead of Oracle-managed defaults for tighter encryption control.

How You Actually Access It

Object Storage has multiple access paths and they are all legitimate. Which one you use depends on what you are doing.

The OCI Console covers everything for manual work: creating and managing buckets, uploading files, configuring PARs, enabling versioning, setting lifecycle rules. It is clean and the bucket detail page surfaces all the relevant settings in one place.

The OCI CLI is better for scripting and bulk operations, uploading directories, syncing files, automating bucket management as part of a deployment pipeline. It supports parallel uploads and the full API surface.

For application development, Oracle provides SDKs for Python, Java, Go, TypeScript, .NET, and Ruby. And critically: Object Storage supports the Amazon S3-compatible API, which means any tool or library built for AWS S3 will work with OCI Object Storage with minimal configuration changes. If you are migrating from AWS or building multi-cloud tooling, this matters a lot.

Monitoring and Cost Visibility

OCI publishes a set of Object Storage metrics through the Monitoring service: storage usage per bucket, request rates, data transfer volumes. You can set up alarms on these metrics (for example: alert me when a bucket exceeds 500 GB), which are then routed through OCI Notifications to email, Slack, PagerDuty, or whatever you have connected.

For cost tracking, OCI generates daily usage reports as CSV files, automatically stored in an Object Storage bucket. These break down resource usage across your tenancy and feed into invoice reconciliation. The fact that the cost reports themselves live in Object Storage is a bit recursive, but it means you can automate ingestion of those reports using the same tools you use to interact with any other bucket.

What the Service Does Quietly in the Background

A lot of what makes Object Storage worth using is not visible in the console. It is what Oracle handles automatically. Data is stored with redundancy across multiple servers within the region. The service performs continuous integrity checking using checksums and repairs any corruption it detects without requiring any action from you. Reads use strong consistency, meaning you always get the latest version of an object. There is no eventual consistency lag to worry about when writing an object and then immediately reading it back.

None of this requires configuration. It is part of what you are getting by using the managed service rather than running your own storage infrastructure.

The documentation is actually good

Oracle's Object Storage docs are detailed and well-organized. The bucket management page and the storage tiers overview are the best places to start if you want to go deeper than this post.
