The problem

What travels across your network is probably readable

Most people think about database security in terms of who can log in and what they can access. That is important, but it only covers half the picture. The other half is what happens to data while it is in motion, traveling from your application across the network to Oracle Database and back again.

Without encryption, that journey happens in plain text. Anyone with access to the network sitting between your application and your database can read every query, every result set, every row returned. This is not a theoretical risk. It is a straightforward packet capture away from becoming very real.

When a query runs against an unencrypted Oracle database, the results travel across the wire as readable text. Names, emails, salaries, addresses. All of it visible to anyone watching the network.

Oracle Database Native Network Encryption (NNE) is the product built to close this gap. It encrypts all data traveling between Oracle clients and Oracle Database servers at the network layer, using industry-strength algorithms, without requiring any changes to your application code or your client installations.

NNE is part of Oracle Net Services, the communication layer built into every Oracle Database installation. That means it is already there. You do not need to install anything extra. You just need to configure it.

How the product works

Encryption built into Oracle Net Services

Oracle Net Services is the layer that manages every connection between an Oracle client and an Oracle Database server. NNE lives inside this layer, which gives it a key advantage: it protects the full connection from the moment it is established, before any SQL is ever sent.

Worth knowing Oracle Database servers and clients default to ACCEPTED mode, meaning encryption turns on automatically the moment either side requests it. You can secure all connections by configuring only the server side.

When a client connects to an Oracle Database server, the two sides go through a negotiation. Each side advertises which encryption algorithms it supports. The server picks the first algorithm in its list that the client also supports, and that algorithm is used for the entire session. If the server is configured to require encryption and no common algorithm is found, the connection is refused.

The session key that powers the encryption is generated fresh for every connection using the Diffie-Hellman key agreement algorithm. Diffie-Hellman is a well-proven method that allows two parties to establish a shared secret over an insecure channel without ever transmitting that secret directly. Once both sides have the shared secret, they combine it with the session key to produce the final encryption key. Compromising one session's key reveals nothing about any other session.

The Advanced Encryption Standard at the core

Oracle NNE uses the Advanced Encryption Standard, known as AES, as its primary encryption algorithm. AES is the Federal Information Processing Standard (FIPS) approved algorithm used by the US government to protect sensitive and classified data. It is the same algorithm at the heart of every serious security product in use today.

Oracle supports three AES key lengths: 128-bit, 192-bit, and 256-bit. All three operate in outer Cipher Block Chaining mode, which means each encrypted block depends on all blocks that came before it. This design makes block replay attacks, where an attacker captures and retransmits encrypted blocks to manipulate the session, computationally infeasible.

AES key lengths in Oracle NNE

AES128 provides solid protection and has the lowest performance overhead. AES192 is a middle ground. AES256 uses a 256-bit key and is the strongest option available. For any new deployment, AES256 is the right choice. Oracle recommends listing it first in your configuration so it takes priority during algorithm negotiation.

All AES variants in Oracle NNE are FIPS 140-2 compliant, which is a requirement in regulated industries including healthcare, finance, and federal government.

Data integrity

Encryption alone is not the whole story

Encrypting network traffic means an attacker cannot read what is being transferred. But Oracle NNE goes a step further and also protects against something encryption cannot handle on its own: data modification in transit.

Oracle's documentation describes two specific attack types that NNE's data integrity feature is designed to prevent. A data modification attack is when someone intercepts traffic, changes the contents, and retransmits it. Think of a database update carrying a $100 transaction that gets intercepted and altered to $10,000. A replay attack is when a valid packet is captured and retransmitted repeatedly. A $100 bank withdrawal that gets replayed ten times becomes a $1,000 loss.

NNE addresses both of these by computing a cryptographic checksum over every message before it is sent. The receiving side recomputes the checksum independently and compares. If a single byte was changed during transit, the checksums will not match and the problem is detected immediately.

◆

Data integrity in NNE operates completely independently from encryption. You can configure checksumming without enabling encryption, or turn on encryption without checksumming. In production environments, enabling both together gives you the most complete protection.

Oracle supports SHA-256, SHA-384, and SHA-512 for checksumming. These are the SHA-2 family of hash functions and Oracle recommends them for all deployments. SHA-1 and MD5 are still present in the product for backward compatibility with older systems but Oracle has deprecated them. Dedicated attacks against MD5 and SHA-1 are practical on modern hardware, so there is no good reason to use them in a fresh configuration.

Configuration

Four settings that control the entire product

The full configuration for Oracle NNE lives in a single file called sqlnet.ora, located in your Oracle Net Services configuration directory. This file controls behavior on both the server and client sides independently. There is no GUI required, no service to deploy, and no restart of the database itself in most cases.

Oracle gives you four possible values for each encryption and checksum parameter. They represent how strongly each side of a connection wants or insists on protection.

REJECTED
This side refuses to encrypt, even if the other side requires it. If the other side is REQUIRED, the connection fails.
ACCEPTED
This side will encrypt if the other side wants it, but will not initiate it. This is the out-of-the-box default for both server and client.
REQUESTED
This side wants encryption and will initiate it. But if the other side cannot do it, the connection continues unencrypted rather than failing.
REQUIRED
This side demands encryption. If the other side cannot or will not encrypt, the connection is refused entirely.

The practical path for rolling out NNE in a production environment is to start with REQUESTED on the server. This immediately enables encryption for every client that supports it, which is essentially every modern Oracle client, while still allowing any legacy clients that do not to keep working. Once you have confirmed that everything is running correctly, you move to REQUIRED, which closes the door on any unencrypted connection.

Because Oracle clients and servers both default to ACCEPTED, setting the server to REQUESTED is enough to turn on encryption for your entire fleet of Oracle clients without touching a single client machine. That is a meaningful operational advantage compared to alternatives that require per-client certificate management.

sqlnet.ora — server configuration
-- File location: $ORACLE_HOME/network/admin/sqlnet.ora

SQLNET.ENCRYPTION_SERVER = REQUESTED
SQLNET.ENCRYPTION_TYPES_SERVER = (AES256, AES192, AES128)

SQLNET.CRYPTO_CHECKSUM_SERVER = REQUIRED
SQLNET.CRYPTO_CHECKSUM_TYPES_SERVER = (SHA512)

After applying the configuration, you can confirm that encryption is active by querying the network service banner for any active session. An encrypted session shows additional rows in the output that were not there before.

Session encryption check output
 SID  USERNAME    AUTH_TYPE   NETWORK_SERVICE_BANNER

The presence of the AES256 Encryption service adapter row confirms the session is fully encrypted. Before NNE is enabled, only the plain TCP/IP adapter row appears. After enabling it, both the encryption and checksumming rows show up alongside it.

What it actually protects

The difference between encrypted and unencrypted traffic

The most concrete way to understand what Oracle NNE does is to look at what network traffic actually contains when the product is off versus when it is on.

Without encryption, a packet capture of Oracle SQL Net traffic running against a table like DEMO_HR_EMPLOYEES will yield full, readable data. Names, email addresses in recognizable formats, salary figures, physical addresses. Every row the database returns passes across the network in a form anyone can read.

The application received its data correctly. The database ran the query normally. The connection worked exactly as before. The only thing that changed is what the network saw, and what the network saw is now completely opaque.

This protection extends to the Glassfish application layer and JDBC thin client connections as well. Oracle NNE works at the Oracle Net Services level, which means it covers any client that uses Oracle's networking stack, including JDBC thin clients used by Java applications, regardless of whether those clients are configured individually.

◆

Connection pool behavior is worth planning around. Applications that create a new connection per query, like many Java web applications using JDBC thin client, pick up the new encryption settings immediately. Applications with long-lived persistent connection pools may need a restart to establish new encrypted connections under the updated configuration.

NNE vs TLS

Oracle gives you two paths to network encryption

Oracle SQL Net traffic can be encrypted in two different ways. The first is Native Network Encryption, the Oracle-proprietary approach we have been discussing. The second is Transport Layer Security, or TLS, which is the industry-standard protocol used everywhere from web browsers to financial systems.

Both encrypt data in motion. But they differ significantly in setup complexity, operational overhead, and what additional security properties they provide.

Consideration Native Network Encryption Transport Layer Security
Setup Edit sqlnet.ora on the server Requires certificates on server and each client
Client changes required Usually none Yes, every client
Certificate management Not needed Required, certificates expire
Protocol type Oracle proprietary Industry standard
Server authentication Not included Yes, prevents impersonation attacks
Can be used for DB user authentication No Yes
Ideal for Internal Oracle environments, rapid rollout Internet-facing systems, compliance-mandated deployments

NNE is the right choice for most internal Oracle environments. It takes minutes to configure, requires no certificate infrastructure, and handles the client side automatically because Oracle clients default to accepting encrypted connections. TLS becomes the better choice when your environment requires server authentication as a protection against impersonation attacks, or when a compliance framework explicitly mandates a standard protocol like TLS over a proprietary one.

Oracle also supports running both at the same time. If some database users connect over TCP with password authentication while others connect over TCPS with TLS certificates, the IGNORE_ANO_ENCRYPTION_FOR_TCPS parameter lets the database handle both connection types concurrently without conflict.

Algorithm hygiene

Strong configuration means more than just turning it on

Enabling NNE is the first step. Making sure you are using strong algorithms is the second. Oracle has deprecated DES, 3DES, RC4, and MD5 in the current release. These algorithms are still present in the product for backward compatibility with very old client versions, but they should not be part of any new configuration. They have known weaknesses and practical attacks exist against several of them.

Oracle released a security patch, documented in My Oracle Support note 2118136.2, that strengthens the NNE implementation and adds two parameters that give administrators control over whether weak algorithms are allowed at all. Once this patch is applied and your clients are updated, you can set those parameters to ensure no connection can fall back to a weak algorithm regardless of what a client requests.

Maximum security configuration in sqlnet.ora
-- Server: require AES256 and block weak crypto from any client
SQLNET.ENCRYPTION_SERVER             = REQUIRED
SQLNET.ENCRYPTION_TYPES_SERVER       = (AES256)
SQLNET.CRYPTO_CHECKSUM_SERVER        = REQUIRED
SQLNET.CRYPTO_CHECKSUM_TYPES_SERVER  = (SHA512)
SQLNET.ALLOW_WEAK_CRYPTO_CLIENTS     = FALSE

-- Client: mirror the server and block weak crypto to any server
SQLNET.ENCRYPTION_CLIENT             = REQUIRED
SQLNET.ENCRYPTION_TYPES_CLIENT       = (AES256)
SQLNET.CRYPTO_CHECKSUM_CLIENT        = REQUIRED
SQLNET.CRYPTO_CHECKSUM_TYPES_CLIENT  = (SHA512)
SQLNET.ALLOW_WEAK_CRYPTO             = FALSE

The ALLOW_WEAK_CRYPTO = FALSE setting is what turns the algorithm restriction from a preference into an enforcement. Without it, a client could potentially negotiate the session down to a weaker algorithm even if you listed only strong ones. With it set to false on both sides, any attempt to use a deprecated algorithm produces an ORA-12269 error at the server or an ORA-12268 error at the client, and the connection is refused.

◆

Sequence matters here. Update all clients before setting ALLOW_WEAK_CRYPTO_CLIENTS to FALSE on the server. If you enforce it server-side before clients are patched, those unpatched clients will lose database connectivity immediately with an ORA-12268 error. Patch clients first, then harden the server.

Bottom line

A product that is genuinely simple to deploy and genuinely important

Oracle Database Native Network Encryption is one of those security features that looks more complicated than it is. The underlying technology, AES encryption and SHA-2 checksumming over a Diffie-Hellman negotiated session key, is serious cryptography. But the operational experience of turning it on is as simple as editing a text file.

What it gives you in return is significant. Every query, every result set, every login credential, every data packet traveling between your Oracle applications and your Oracle Database becomes unreadable to anyone intercepting the network. The encryption is transparent to your application. Your SQL does not change. Your connection strings do not change. Your users do not notice anything different. The only thing that changes is that the wire is now protected.

Having database authentication in place but leaving network traffic unencrypted is like putting a lock on your door while leaving the windows open. Both matter.

Oracle Database 19c ships with NNE already built in. The server and clients default to ACCEPTED mode, which means encryption can be turned on for your entire Oracle client fleet by changing a single parameter on the server side. No agents to deploy, no certificates to issue, no application code to touch.

Quick reference for a production rollout

1. Set SQLNET.ENCRYPTION_SERVER = REQUESTED in the server sqlnet.ora. This immediately enables AES encryption for all connecting Oracle clients without requiring any client-side changes.

2. Add SQLNET.ENCRYPTION_TYPES_SERVER = (AES256) to ensure the strongest algorithm is selected during negotiation.

3. Enable checksumming with SQLNET.CRYPTO_CHECKSUM_SERVER = REQUIRED and SQLNET.CRYPTO_CHECKSUM_TYPES_SERVER = (SHA512) to add data integrity protection on top of encryption.

4. Verify by checking the network service banner on an active session. Look for the AES256 Encryption service adapter row.

5. Once confirmed, step up to REQUIRED and apply Oracle patch 2118136.2 to block weak algorithm fallback entirely.
